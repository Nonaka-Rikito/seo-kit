# SEO 記事変更指示書スキル

公開済みのブログ記事に対して、Googleドキュメントの「提案モード」に相当する変更指示書を生成します。記事を全文書き直すのではなく、「元のテキストのどこをどう変えるか」を番号付きの変更一覧で出力します。

## 引数

`$ARGUMENTS` に対象記事のURLを指定してください。

```
/seo-change-brief https://example.com/blog/article/
```

複数記事を一括処理する場合は、指示書ファイルのパスを渡します。

```
/seo-change-brief /path/to/rewrite-instructions.md
```

## 出力形式

各変更を以下の表形式で番号順に出力します。

### 既存テキストの修正・追記（変更後（追記）欄）

```
### #N｜変更箇所の名前
| 項目 | 内容 |
|------|------|
| 変更種別 | 🔴 古い情報 / ✏️ 追記・改善 / 🔗 リンク確認 |
| 場所 | H2「○○」セクション など |
| 変更前（原文） | 元のテキストをそのまま引用 |
| 変更後（追記） | 変更後のテキスト |
| 理由 | なぜ変更が必要か |
```

### 新規セクションの追加（変更後（新規追加）欄）

```
### #N｜追加するセクション名
| 項目 | 内容 |
|------|------|
| 変更種別 | ✏️ 追記・構成改善 |
| 場所 | 「○○」セクションの後 ／ リード文直後 など ※挿入位置を必ず明記 |
| 変更前（原文） | 該当箇所なし（新規追加） |
| 変更後（新規追加） | 「○○」セクションの後に、以下のH2見出しと本文を追加する |
| **追加する見出し：** | ## ○○ |
| **追加する本文：** | （100〜150文字の結論ベース文章。必ず記載すること） |
| **追加する表：** | ※表が必要な場合のみ。本文の後に配置する |
| 理由 | なぜ追加が必要か |
```

変更種別の定義：
- 🔴 **古い情報・事実誤り**：日付・数値・名称・仕様が現在と異なるもの
- ✏️ **追記・構成改善**：SEO・品質・読者体験を向上させるための追記や構成変更
- 🔗 **リンク確認**：リンク切れや移転の可能性があるもの

最後に**優先度まとめ表**を付ける。

```
| 優先 | # | 変更内容（一言） | 対応時間目安 |
```

## 実行手順

1. `$ARGUMENTS` がURLの場合：WebFetchで記事本文を取得する
2. `$ARGUMENTS` がファイルパスの場合：そのファイルを読み込み、記事URLを特定してWebFetchで取得する
3. 取得した内容をもとに、以下の観点で変更点を洗い出す（「チェック観点」参照）
4. 変更指示書を「出力形式」に沿って生成する
5. ファイルに保存する（パスは「ファイル保存先」参照）
6. チャットには保存先パスと変更件数・優先度サマリーのみ返す

## チェック観点

記事を分析する際は以下を必ずチェックする。

### 🔴 事実・情報の正確性
- **プラットフォーム名称の変更**：Twitter → X（旧Twitter）、Facebook広告 → Meta広告 など
- **文字数制限の変更**：Twitterの「140文字」→「280文字」（2017年11月〜）
- **ユーザー数・市場データ**：引用データの年月を確認し、3年以上前のものにはその旨を注記
- **料金・費用相場**：ツール料金、広告相場、代理店手数料などは変動が大きい
- **機能・仕様の変更**：E-A-T → E-E-A-T（2022年12月〜）、AMP推奨の縮小、iOS14以降の計測変化など

### ✏️ 構成・SEO改善
- **タイトルに年号がない**：「最新版」だけでは鮮度が伝わらない。「2025年最新版」等に変更
- **検索意図とのズレ**：タイトルが「どれを選ぶべき？」なのに選び方の指針がない、など
- **目的別・比較表の不足**：選択肢が多い記事で「使い分けの基準」が示されていない
- **新機能・新概念の未掲載**：記事公開後に登場した主要な機能や変化が反映されていない
  - 例：Advantage+オーディエンス、ブランドリフト計測、生成AIによるクリエイティブ制作 など
- **E-E-A-T（経験・専門性）の強化**：「一般的に言われている」で終わっている箇所を具体的な事例・数値・実績に置き換え

### 🔗 リンク・出典の確認
- **外部リンクのドメイン移転リスク**：about.fb.com / newsroom.fb.com、Twitterの公式URL など
- **出典の年月**：数値の引用元が古く、現在と異なる可能性のあるもの
- **内部リンクの存在確認**：関連記事として言及されているページが現在も存在するか

## WebFetchの制限への対応

WebFetchは原文を要約して返すことがある。その場合：
- 「変更前（原文）」欄に `※近似` と明記する
- テキストの完全な確認は担当者に委ねる旨を末尾の確認リストに記載する

## ファイル保存先

デフォルト：`C:/Users/rikit/Projects/malna/output/article-changes-{連番}-{記事スラッグ}.md`

クライアントや保存先が異なる場合は `$ARGUMENTS` で明示する。

## 新規追加セクションのベストプラクティス

新規セクション（`変更後（新規追加）`）を生成する際は以下を必ず守ること。

### 1. 挿入位置を必ず明記する

`変更後（新規追加）` 欄の冒頭に、どのセクションの後に挿入するかを日本語で明示する。

**NG例（位置不明）**：
```
| 変更後（新規追加） | 以下のH2見出しと本文を追加する |
```

**OK例（位置明記）**：
```
| 変更後（新規追加） | 「ターゲティングの設定方法」セクションの後に、以下のH2見出しと本文と表を追加する |
```

複数の候補がある場合は `「○○」または「△△」セクションに` と並記してよい。

### 2. 見出しの直後には必ず本文を入れる

`**追加する見出し：**` がある場合、その直後に必ず `**追加する本文：**` を記載する。**表だけで本文なしはNG**。

- 本文の分量：**100〜150文字**
- 文体：結論ベース（「〜です。〜しましょう。」など断言調）
- 本文 → 表の順番で配置する

**NG例（本文なし）**：
```
| **追加する見出し：** | ## 広告フォーマット比較 |
| **追加する表：** | （表） |
```

**OK例（本文あり）**：
```
| **追加する見出し：** | ## 広告フォーマット比較 |
| **追加する本文：** | Meta広告では目的に応じて最適なフォーマットを選ぶことが成果を左右します。〜（100〜150文字） |
| **追加する表：** | （表） |
```

### 3. 「変更後（新規追加）」欄に内容を要約する

`変更後（新規追加）` 欄には挿入位置 + 追加内容の種類を明示する。

- 本文のみ：`〜セクションに、以下のH2見出しと本文を追加する`
- 本文＋表：`〜セクションに、以下のH2見出しと本文と表を追加する`
- H3見出し＋本文＋表：`〜セクションの末尾に、以下のH3見出しと本文と表を追加する`

## 注意事項

- 記事を全文書き直すのではなく、**変更箇所の特定と差し替え文の提示**に徹すること
- 元の記事にない情報・数値を変更後テキストに追加しない（追記は明示した上で行う）
- 変更件数は多すぎると対応負荷が高くなるため、🔴高優先を確実に押さえ、✏️は影響の大きいものに絞る
- このコマンドは既存公開記事のメンテナンスに使用する。新規記事制作は `/seo-writing` を使うこと
