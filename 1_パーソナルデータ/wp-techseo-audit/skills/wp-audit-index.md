# WordPress インデックス状況確認

Google Search Console を通じてサイトのインデックス状況を確認し、問題ページの再クロールをリクエストします。

## 引数

$ARGUMENTS にクライアントIDを指定してください。（例: `naimono`）

## 実行手順

あなたはテクニカルSEOの専門家です。GSC MCP サーバーを活用してインデックス状況を確認してください。

### Step 1: クライアント設定の確認

`config/clients.json` からクライアントの `gsc_site_url` を取得します。

### Step 2: インデックスカバレッジの確認

GSC MCP ツールを使って以下を確認：

1. **インデックス登録済みページ数** — GSC の「ページ」レポート相当
2. **インデックス未登録ページ** — クロール済みだがインデックスされていないページ
3. **エラーページ** — サーバーエラー、リダイレクトエラー等

利用可能な GSC MCP ツール:
- `inspect_url_enhanced` — 個別URLのインデックス状況を詳細確認
- `batch_url_inspection` — 複数URLの一括確認
- `check_indexing_issues` — インデックス問題の一覧
- `get_performance_overview` — パフォーマンス概要

### Step 3: 問題URLの特定

クロール監査の結果（もしあれば）と照合し、以下の問題を特定：

- **インデックスされるべきだがされていないページ**
  - canonical 設定ミス
  - noindex タグ誤設定
  - サーバーエラー
  - リダイレクト問題

- **インデックスされるべきでないがされているページ**
  - 重複ページ
  - 管理ページ
  - パラメータ付きURL

### Step 4: サイトマップとの整合性チェック

wp_reader.py でサイトマップURLを取得し、GSC の認識状況と比較：

```bash
"C:\Users\rikit\Projects\1_パーソナルデータ\seo-machine\.venv\Scripts\python" scripts/wp_reader.py --client $ARGUMENTS --action sitemap
```

- サイトマップに含まれるがインデックスされていないURL
- インデックスされているがサイトマップに含まれないURL

### Step 5: 検索パフォーマンスの確認

GSC MCP の `get_search_analytics` または `get_advanced_search_analytics` で：
- インプレッションはあるがクリックがないページ
- 順位が下落傾向のページ
- 新規インデックスされたページのパフォーマンス

### Step 6: レポート生成

```markdown
# インデックス状況レポート

**クライアント**: [名前]
**GSCサイト**: [URL]
**確認日時**: [日時]

## サマリー

| 指標 | 値 |
|------|-----|
| インデックス登録済み | N ページ |
| インデックス未登録 | N ページ |
| エラー | N ページ |
| サイトマップURL数 | N |

## インデックス問題

### 要対応: インデックスされるべきだがされていないページ

| URL | 原因 | 推奨対応 |
|-----|------|---------|
| ... | ... | ... |

### 注意: インデックスされるべきでないページ

| URL | 現状 | 推奨対応 |
|-----|------|---------|
| ... | ... | ... |

## サイトマップ整合性

- サイトマップにあるがインデックス外: N件
- インデックスにあるがサイトマップ外: N件

## 検索パフォーマンス要注意ページ

| URL | インプレッション | クリック | CTR | 平均順位 | 問題 |
|-----|-----------------|---------|-----|---------|------|
| ... | ... | ... | ... | ... | ... |

## 推奨アクション

1. ...
2. ...
```

レポートは `reports/{client}_index_{date}.md` に保存。

## 注意事項

- GSC のデータは通常2-3日遅延がある
- URL検査APIには日次レート制限がある（大量一括検査は避ける）
- インデックス登録リクエストは1日あたりの上限がある
- 修正適用後は48-72時間後に再確認するのが適切
