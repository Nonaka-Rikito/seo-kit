## 1. 現状の課題と問題点の整理

### (a) なぜ今のやり方に問題があるのか

- **AIツールの進化スピードに個人の作業方法が追いついていない**
  ChatGPTやブラウザベースのAIを「チャット」として使うだけでは、人間が情報のコピペ・加工・転記をする部分がボトルネックとして残り続ける。

- **個人がバラバラにツールを使っているため、ナレッジが属人化する**
  各自がどんなプロンプトを使っているか、どうやって品質を上げているかが共有されず、組織として蓄積されない。

- **「AIを使っている」≠「仕事が速い」**
  AIをどう使うか、どのレイヤーで使うかが重要。チャットに指示を出して結果をコピペしているだけでは、AIの能力を10%も引き出せていない。

- **2月11日の導入報告から4日経つが、まだ全員が実際に触れていない可能性がある**
  導入の意義を理解していても、実際に手を動かさなければ何も変わらない。

### (b) どのように変えていくべきか

**結論: すべてのアウトプット（文章、リサーチ、企画、コード、資料）をClaude Code経由で出す。**

「Claude Codeを通して出す」とは具体的に以下を意味する：

1. ターミナルで `claude` を起動する
2. 作りたいものを自然言語で指示する
3. Claude Codeが社内のルール（CLAUDE.md）・参照データ（input/）・MCP連携を活用して出力する
4. 人間がレビュー・修正指示を出す
5. 最終成果物をファイルとして保存する（チャットに垂れ流さない）

**メリット:**

- **ルールの自動適用**
  CLAUDE.mdに蓄積されたルール・スタイル・フィードバックが自動で反映される。個人がいちいち「どう書くんだっけ？」と思い出す必要がない。

- **MCP連携でコピペ不要**
  Slack/Calendar/Notion/Gmail/Chatwork/GitHub/Ahrefsなどに直接アクセスできる。情報を手動でコピペする時間がゼロになる。

- **成果物がファイルとして残る**
  Gitリポジトリに保存されるため、バージョン管理と再利用が可能。Slackのスレッドに埋もれて探せなくなることがない。

- **個人の暗黙知がチームの資産になる**
  良いプロンプトや参照データがCLAUDE.mdやinput/に言語化され、他のメンバーも同じクオリティを出せるようになる。

## 2. チームでの最大アウトプットに向けた懸念事項

### (a) チーム運用におけるGit関連知識の必要性

Claude Codeをチームで運用するには **Gitの基本知識が必須**。

**なぜGitが必要か:**

- malna-opsリポジトリを全員が共有している
- CLAUDE.mdやinput/のルールファイルを複数人が更新する可能性がある
- 成果物（output/drafts/）の管理にGitが使われている
- 「誰がいつ何を変えたか」がGitで追跡される

**最低限必要な知識:**

| コマンド | 用途 |
|----------|------|
| `git pull` | 最新を取得する |
| `git add` / `git commit` | 自分の変更を記録する |
| `git push` | 共有リポジトリに反映する |
| `git status` | 現状を確認する |
| コンフリクト対処 | 基本：高橋に相談 |

**学習コスト:**

- 1〜2時間のハンズオンで基本操作は習得可能
- Claude Code自身にGit操作を教えてもらうこともできる（「Gitの使い方を教えて」と聞けばいい）
- 最初は高橋がペアで一緒にやる

**今後のステップ:**

- 来週中（2/17〜2/21）にGitの最低限ハンズオンを実施する

### (b) 個人がローカルでコツコツやることの限界

個人PCでClaude Codeを使うだけでは組織的な効果は半減する。

**問題点:**

- 個人ローカルのCLAUDE.mdにしかルールがないと、他のメンバーが同じ品質を出せない
- 各自のプロンプトのノウハウが共有されない
- 成果物がSlackやNotionのコメントに分散し、再利用できない
- 「あの人がいないと分からない」状態が温存される

**解決策:**

- **malna-opsを「チームの知識ベース」として全員が参照・更新する**
- **input/以下のルールファイルは全員が改善提案できる**（PRまたは高橋に口頭で伝える）
- **成果物は必ずoutput/drafts/にファイルとして保存する**
- **良いプロンプトやワークフローはdocs/に言語化して残す**

## 3. 明日からの具体的な運用ルール

### 対象者

- 正社員全員（3名）
- 役員

### 必須ルール

1. **すべてのアウトプットはClaude Codeを通す**
   - 文章作成、リサーチ、企画立案、コード作成、メッセージ返信の下書き
   - 例外: 緊急の1〜2行のSlack返信、電話メモなど即時性が求められるもの

2. **成果物はファイルとして保存する**
   - 保存先: `output/drafts/` またはタスクに応じた適切なディレクトリ
   - ファイル名規則: `[用途]_[テーマ].md`（例: `企画書_新規LP制作.md`）

3. **毎日最低1回はClaude Codeを起動して業務に使う**
   - 「何に使えばいいか分からない」場合は高橋に相談

4. **分からないことはClaude Code自身に聞く**
   - 「Gitの使い方を教えて」「この操作をやりたいんだけど」→ Claude Codeがガイドしてくれる

5. **良いプロンプトや使い方を発見したらチームに共有する**
   - Slack #dev チャンネルまたはdocs/に記録

### 推奨ルール

- **Agent Teamsを週1回は使う**（企画・戦略の叩き台作成に最適）
- **Cursorでのコーディング時もClaude Codeの知見を活用する**
- **フィードバックがあればinput/feedback_log.mdに追記する**
