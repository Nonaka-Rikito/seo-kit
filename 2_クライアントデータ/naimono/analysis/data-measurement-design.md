# データ計測設計書 — クリエイティブ変更前後の効果測定

**作成日**: 2026年2月17日
**対象サイト**: jo-katsu.com
**GA4 Property ID**: 356839446
**目的**: クリエイティブ変更が与えた因果的効果を正確に計測する

## エグゼクティブサマリー

本設計書は、jo-katsu.comにおけるクリエイティブ（バナー・記事リライト等）変更の効果を**統計的に正確**かつ**バイアスを最小化**して計測するための技術仕様書です。

**3つの設計原則**:
1. **因果推論の担保**: DID（差分の差分）分析により外部要因を分離
2. **データ品質第一**: 欠損・遅延・定義のブレを前提とした設計
3. **バイアスの明示**: 完全排除は不可能。残存バイアスを常に記録

**最重要推奨事項**:
- **P0**: GSC権限の即時取得（CTR計測に必須）
- **P1**: 対照群（control group）データの同時取得（DID分析の基盤）
- **P1**: Phase 2/3レポート（45日後・90日後）の実装

## 1. データアーキテクチャ

### データフロー
```
[記事変更] → [Google Forms入力] → [GAS: Before取得]
    ↓
GA4 API + GSC API + HTMLスクレイピング
    ↓
[Spreadsheet保存] → [17日後: After取得] → [DID分析] → [Slack通知]
```

### データソース

| ソース | メトリクス | API | 遅延 |
|--------|-----------|-----|------|
| GA4 | セッション, PV, エンゲージメント, CV | `AnalyticsData.Properties.runReport` | ~24h |
| GSC | クリック, 表示, CTR, 順位 | `Webmasters.Searchanalytics.query` | **2-3日** |
| HTML | OG画像, バナー画像 | `UrlFetchApp.fetch` | なし |

## 2. 正確な計測を実現する手法

### 2.1 計測期間の設計

| パラメータ | 設定値 | 根拠 |
|-----------|-------|------|
| BEFORE_DAYS | 14日 | 曜日効果2周期分 |
| AFTER_DAYS | 14日 | Before と同一 |
| GSC_DELAY_BUFFER | 3日 | GSC遅延の安全マージン |

**推奨改善**:
- Phase 1: 変更後17日（現行）
- Phase 2: 変更後45日（順位安定後）
- Phase 3: 変更後90日（長期効果）

### 2.2 対照群（Control）の設計 ⭐最重要

**なぜ必要か**: Before/After比較だけでは以下を分離できない
- Googleアルゴリズム変更
- 季節需要の変動
- AI Overview展開率の変化
- 競合の動き

**対照群の選定基準**:
1. 同一カテゴリ
2. 類似トラフィック規模
3. 類似順位帯
4. 対象期間中に変更なし

**DID（差分の差分）計算**:
```
真の効果 = (処理群のAfter - Before) - (対照群のAfter - Before)
```

**具体例**:
| 変更対象 | 対照群候補 | 理由 |
|---------|-----------|------|
| /campus/10673 (MBTI一覧) | /campus/5196 (性格一覧) | 同カテゴリ、類似規模 |
| /campus/5504 (リクルートスーツ) | /campus/22136 (履歴書封筒) | 就活準備、Top3 |

### 2.3 日次データ粒度の計測

**現状の問題**: 期間合計値のみ → 変化点検出不可

**推奨**: GA4 API の `dimensions` に `date` を追加
```javascript
dimensions: [{ name: 'date' }, { name: 'pagePath' }]
```

**メリット**:
- 時系列トレンドの可視化
- 変化点検出（いつから効果が発現したか）
- 異常値の検出

## 3. データ品質チェックリスト

### 3.1 完全性（Completeness）

| 項目 | 確認方法 | 対処 |
|------|---------|------|
| GA4データ欠損 | `noData: true` | エラー通知 + 再実行 |
| GSCデータ欠損 | 同上 | 「GSCデータなし」明示 |
| スクロール深度欠損 | GTMイベント確認 | 事前確認必須化 |

### 3.2 一貫性（Consistency）

| 項目 | リスク | 対処 |
|------|-------|------|
| pagePath正規化 | クエリパラメータ混在 | `extractPagePath()` 実装済 |
| タイムゾーン | 日付ズレ | Asia/Tokyo統一済 |
| チャネルグループ | 定義変更 | Organic Search固定済 |

### 3.3 鮮度（Freshness）

| ソース | 遅延 | 影響 |
|--------|-----|------|
| GA4 | ~24h | beforeEnd = changeDate -1 で対応 |
| GSC | 2-3日 | GSC_DELAY_BUFFER: 3 で対応 |

## 4. バイアス排除の具体的手法

### 4.1 セレクションバイアス
**リスク**: 改善が見込まれる記事のみを選定 → 改善バイアス
**対処**: 対照群比較（DID）、レポートに選定基準を明記

### 4.2 タイミングバイアス
**リスク**: 就活サイトの季節性（3月ピーク）→ リライト効果と混同
**対処**: DID（季節要因は差分で相殺）、前年同月比併記

### 4.3 観測者バイアス
**リスク**: 担当者の確証バイアス
**対処**: 閾値事前設定（±5%）、自動判定の機械的適用

### 4.4 サバイバーシップバイアス
**リスク**: 改善事例のみ報告、悪化事例は無視
**対処**: 全変更の記録、月次サマリー、悪化アラート

### 4.5 回帰効果（平均回帰）
**リスク**: 急落記事は自然回復 → リライト効果と誤認
**対処**: DID（対照群も同様に減少なら回帰効果は相殺）

### 4.6 複数施策の交絡
**リスク**: 同時実施でどの施策が効果か不明
**対処**: 段階的実施、施策タグ記録、現実的な割り切り

## 5. A/Bテストの要否判断

### 結論: 本ケースでは不要（実施すべきでない）

**理由**:
1. SEO施策にA/Bは不適合（クローキングリスク）
2. サンプルサイズ不足（数週間〜数ヶ月必要）
3. 緊急性（リカバリー優先、機会損失回避）

### 代替手法: 準実験デザイン
1. DID（差分の差分）
2. 時系列介入分析
3. 段階的ロールアウト

## 7. 既存GASシステムへの改善提案

### 7.1 対照群データの同時取得

**Config.gsに追加**:
```javascript
CONTROL_GROUPS: {
  '/campus/10673': ['/campus/5196', '/campus/8535'],
  '/campus/5504':  ['/campus/22136', '/campus/9346'],
  // ...
}
```

**Main.gsで実装**:
対照群URLに対しても `fetchGA4Data()` / `fetchGSCData()` 実行
→ `periodType: 'Before-Control'` で記録

### 7.2 判定ロジックの強化

1. DID判定: 対照群の変化を差し引いた値で判定
2. 信頼度スコア: 日次データの分散から信頼度を付記
3. GSCデータなし時の調整: GA4のみで判定するモード

## 8. 計測KPIの定義

### 一次KPI（効果判定の主軸）

| KPI | 重み | データソース |
|-----|------|-------------|
| オーガニックセッション数 | 40% | GA4 |
| GSC CTR | 30% | GSC |
| エンゲージメント率 | 20% | GA4 |
| 直帰率（逆指標） | 10% | GA4 |

### 判定基準

| 判定 | 条件 |
|------|------|
| 改善 | 重み付きスコア >= +5% |
| 悪化 | 重み付きスコア <= -5% |
| 変化なし | -5% < スコア < +5% |

## 9. 計測の限界と注意事項

### 9.1 因果推論の限界
- 未観測交絡因子（対象記事だけに特異的な要因）
- SUTVA違反（内部リンクによる波及効果）
- 外的妥当性（一般化の限界）

### 9.2 GSC権限の問題 ⚠️

**影響**: CTR比較不可、順位追跡不可、クエリ分析不可

### 9.3 サンプリングの問題

→ `metadata.samplingMetadatas` 確認、警告付記
