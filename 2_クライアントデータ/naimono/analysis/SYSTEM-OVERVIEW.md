# jo-katsu.com SEO 自動モニタリング＋仮説エンジン — システム全体概要

**作成日**: 2026年2月21日
**対象サイト**: jo-katsu.com（就活情報メディア）
**追加コスト**: 0円〜（GAS + GA4/GSC API は無料枠運用。Ahrefs利用時は契約プランに依存）

---

## 1. このシステムでできること

### A. 週次全ページ自動モニタリング（毎週月曜 AM 9:00）
- jo-katsu.com の**全記事**を GA4 を主軸に取得し、GSC 指標を併合して評価
- 前週比 20%以上の変動がある記事を自動検出
- 検出された記事の GSC データ（順位・CTR・クエリ変動）を詳細取得
- 必要に応じて Ahrefs データを補助的に取得（競合/SERP変化の補足）
- **なぜ変動したか**の因果仮説をルールベースで自動生成
- 仮説に基づく**具体的な施策提案**（優先度・工数付き）
- 施策を**スプレッドシートのキュー**に自動追加
- Slack にレポート送信

### B. バナー差替え Before/After 計測（Googleフォーム送信時）
- フォームに URL を入力 → Before データを自動取得
- 17日後に After データを自動取得し、12指標で比較
- CTA変更が「改善」「悪化」「変化なし」かを判定
- **CTA効果の因果仮説**を自動生成
- 施策提案を自動生成しキューに追加
- Slack に判定レポート送信

### C. 施策キュー管理（スプレッドシート手動操作）
- 提案された施策の承認→実施→検証をスプレッドシート上で管理
- ステータス: `提案 → 承認 → 実施中 → 計測中 → 検証完了 / 却下`

## 4. 仮説エンジン（HypothesisEngine.gs）

### 10ルール一覧

| # | ルールID | 検出条件 | 仮説テキスト例 |
|---|---------|---------|--------------|
| 1 | `RANKING_LOSS` | GSC順位が1.5pt以上悪化 | 検索順位の下落によるトラフィック減少の可能性 |
| 2 | `RANKING_GAIN` | GSC順位が1.5pt以上改善 | 検索順位の改善によるトラフィック増加の可能性 |
| 3 | `CTR_DROP` | CTR 5%以上低下 & 順位横ばい | SERP表示の変化（AI Overview等）によるCTR低下の可能性 |
| 4 | `QUERY_LOSS` | 主要クエリのクリック5%以上減少 | 主要検索クエリ「{query}」でのクリック減少 |
| 5 | `NEW_QUERY` | 新規クエリで5クリック以上 | 新規クエリ「{query}」からの流入開始 |
| 6 | `BOUNCE_SPIKE` | 直帰率が10pt以上上昇 | コンテンツとユーザー検索意図のミスマッチの可能性 |
| 7 | `SEASONAL` | 就活関連パスで季節変動 | 就活シーズンに連動した季節的な変動の可能性 |
| 8 | `CTA_POSITIVE` | バナー差替え後に複合指標改善 | CTA変更による正の効果 |
| 9 | `CTA_NEGATIVE` | バナー差替え後に複合指標悪化 | CTA変更による負の効果。差し戻しを検討 |
| 10 | `IMAGE_SEARCH` | 画像検索クリック5%以上変動 | 画像検索からのトラフィック変動 |

### しきい値（MonitorConfig.gs で管理）

```
RANKING_CHANGE:       1.5     順位変動（ポイント）
CTR_DROP_RATE:        0.20    CTR低下率 20%（5%ではノイズが多すぎるため）
CTR_POSITION_STABLE:  1.0     「順位横ばい」の範囲
QUERY_CLICK_DROP:     0.50    クエリクリック減少率 50%（主要クエリの大幅減少のみ検出）
NEW_QUERY_MIN_CLICKS: 5       新規クエリ最低クリック数
BOUNCE_SPIKE_PT:      0.10    直帰率急増しきい値 10pt
IMAGE_SEARCH_CHANGE:  0.50    画像検索変動 50%（大幅な変動のみ検出）
```

### 季節性パターン（jo-katsu.com 専用）

| パス | ピーク月 | オフ月 |
|------|---------|--------|
| `/intern/`, `/campus/`, `/recruit/` | 1,2,3,5,6月 | 8,9,11,12月 |
| `/manner/` | 2,3,4月 | 7,8,9月 |

## 5. 施策マッピング（ActionProposer.gs）

### 仮説→施策テーブル

| 仮説 | 施策 | 優先度 | 工数 | スコア |
|------|------|--------|------|--------|
| RANKING_LOSS | コンテンツリライト | 中 | 高 | 0.67 |
| RANKING_LOSS | 内部リンク強化 | 中 | 低 | 2.0 |
| CTR_DROP | タイトル・メタ見直し | 高 | 低 | **3.0** |
| CTR_DROP | 構造化データ追加 | 中 | 中 | 1.0 |
| QUERY_LOSS | 失ったクエリのセクション強化 | 高 | 中 | 1.5 |
| QUERY_LOSS | 関連記事からの内部リンク追加 | 中 | 低 | 2.0 |
| NEW_QUERY | 新規クエリ向けセクション追加 | 中 | 中 | 1.0 |
| BOUNCE_SPIKE | コンテンツ冒頭の改善 | 高 | 中 | 1.5 |
| BOUNCE_SPIKE | 目次・CTA配置の見直し | 中 | 低 | 2.0 |
| SEASONAL | 経過観察 | 低 | 低 | 1.0 |
| CTA_POSITIVE | 成功パターンを横展開 | 中 | 中 | 1.0 |
| CTA_NEGATIVE | 旧CTAに差し戻し | 高 | 低 | **3.0** |
| CTA_NEGATIVE | 別デザインでA/Bテスト | 中 | 中 | 1.0 |
| IMAGE_SEARCH | 画像alt・ファイル名最適化 | 中 | 低 | 2.0 |

**スコア** = 優先度(高=3,中=2,低=1) / 工数(低=1,中=2,高=3)。高いほど費用対効果が高い。

## 6. 施策キュー（ExecutionQueue.gs）

### シート構造：「施策キュー」

| 列 | フィールド | 説明 |
|----|----------|------|
| A | キューID | 自動生成（UUID 8桁） |
| B | 提案日 | 自動記入 |
| C | ページURL | 対象記事URL |
| D | 施策タイトル | 施策名 |
| E | 施策詳細 | 具体的な施策内容 |
| F | ソース | 「モニタリング」/「バナー差替え」 |
| G | 仮説タイプ | ruleId（RANKING_LOSS等） |
| H | 優先度 | 高/中/低 |
| I | 工数 | 低/中/高 |
| J | スコア | 費用対効果スコア |
| K | ステータス | **手動で変更**（下記ワークフロー参照） |
| L | 備考 | 自由記入 |

### ステータスワークフロー

```
提案（自動）→ 承認（手動）→ 実施中（手動）→ 計測中（手動）→ 検証完了（手動）
                └→ 却下（手動）
```

- **自動で行われるのは「提案」ステータスでの書き込みのみ**
- 承認以降は全てスプレッドシート上で手動操作
- 同じページ+同じ施策が既に提案中の場合は重複スキップ

## 11. セットアップ手順

### 前提条件
- Google アカウント
- jo-katsu.com の GA4 / GSC へのアクセス権
- Ahrefs API キー（任意。補助データ取得を使う場合）
- Slack ワークスペースの管理権限

### 手順

**Step 1: Google フォーム + スプレッドシート作成**
1. Google Forms で新規フォーム作成（上記フィールド構成を参照）
2. フォーム「回答」→ スプレッドシート作成
3. スプレッドシート URL から `SPREADSHEET_ID` をメモ

**Step 2: Slack Incoming Webhook 取得**
1. https://api.slack.com/apps → 「Create New App」
2. 「Incoming Webhooks」→ ON → チャンネルを選択
3. Webhook URL をコピー

**Step 3: Config.gs に本番値を設定**
- `GA4_PROPERTY_ID`, `GSC_SITE_URL`, `SPREADSHEET_ID`, `SLACK_WEBHOOK_URL` を変更
- Ahrefsを使う場合は `AHREFS_API_KEY` を設定し、`MonitorConfig.gs` の `ENABLE_AHREFS_FALLBACK` を有効化

**Step 4: GAS デプロイ**
1. スプレッドシートから「拡張機能」→「Apps Script」
2. 16ファイルを GAS エディタに転記（ファイル名は `.gs` を除いた名前で作成）
3. 「サービス」→ 以下を追加:
   - **Google Analytics Data API** (v1beta) → 識別子: `AnalyticsData`
   - **Google Search Console API** (v3) → 識別子: `Webmasters`
4. `appsscript.json` を表示して以下のスコープを確認:
   ```json
   {
     "oauthScopes": [
       "https://www.googleapis.com/auth/spreadsheets",
       "https://www.googleapis.com/auth/analytics.readonly",
       "https://www.googleapis.com/auth/webmasters.readonly",
       "https://www.googleapis.com/auth/script.external_request",
       "https://www.googleapis.com/auth/forms",
       "https://www.googleapis.com/auth/script.scriptapp"
     ]
   }
   ```

**Step 5: トリガー設定**
1. GAS エディタから `setupWeeklyTrigger()` を1回実行（週次モニタリング用）
2. フォームのトリガー設定: `onFormSubmit` → フォーム送信時

**Step 6: 動作確認**
1. `testWeeklyMonitoring()` を実行 → Slack にレポートが届くか確認
2. `testFormSubmit()` を実行 → Slack に検知通知が届くか確認
3. スプレッドシートに「施策キュー」シートが作成されているか確認

## 12. 運用フロー

### 週次（自動）
1. 毎週月曜 AM 9:00 に自動実行
2. Slack にレポートが届く
3. 施策キューに提案が追加される
4. スプレッドシートで施策を確認し、承認/却下を手動で判断
5. 深い分析が必要な場合は Slack レポートを Claude Code に貼り付け

### バナー差替え時
1. Google フォームに URL + 変更情報を入力
2. 17日後に自動レポート
3. 施策キューに CTA 関連の提案が追加される

### 施策キュー管理
1. スプレッドシートの「施策キュー」を定期確認
2. スコア順（J列）でソートして優先度の高い施策から対応
3. ステータスを手動で更新: 承認 → 実施中 → 計測中 → 検証完了

## 13. トラブルシューティング

| 症状 | 対処 |
|------|------|
| Phase 1 でエラー | Slack にエラー通知。`resetMonitoringState()` を実行してリセット |
| Phase 2 でエラー | `retryFromPhase2()` で Phase 2 のみ再実行 |
| Slack に通知が来ない | `SLACK_WEBHOOK_URL` の確認。GAS エディタのログで確認 |
| GA4 データが空 | `GA4_PROPERTY_ID` の確認。Advanced Service の有効化を確認 |
| GSC データが空 | `GSC_SITE_URL` の確認（末尾スラッシュに注意）。3日遅延は正常 |
| Ahrefs データが空 | `AHREFS_API_KEY` の確認。未設定時は GA4/GSC のみで継続実行 |
| 施策キューに重複 | 同じページ+同じ施策タイトルは自動スキップ。手動で「却下」も可 |
| 全状態リセット | `resetMonitoringState()` を実行 |
| 全トリガー確認 | `listAllTriggers()` を実行 |
| 全トリガー削除 | `deleteAllTriggers()` を実行（注意して使用） |

## 14. 注意事項

- **因果と相関の分離**: 仮説テキストには「可能性」を含め、断定はしない。あくまでルールマッチングの結果
- **人間承認が必須**: 施策キューの自動操作は「提案」のみ。承認以降は必ず人間が判断
- **データ鮮度**: GSC は 3日遅延があるため、「今週」は [today-10 ~ today-4] で計算
- **データソース方針**: GA4/GSC を主データソースとし、Ahrefs は必要時のみ補助的に使用
- **GAS 6分制限**: Phase 1 + Phase 2 に分割して対応済み。フラグ付きページが20件を超えないよう制限
- **深い分析は Claude Code**: ルールベースエンジンは定型パターンの検出のみ。「なぜ他サイトに負けているか」等の深い分析は Slack レポートを Claude Code に貼り付けて依頼
- **しきい値の調整**: 運用しながら `MonitorConfig.gs` の値を調整可能（例: 20%だとノイズが多い → 30%に上げる）
