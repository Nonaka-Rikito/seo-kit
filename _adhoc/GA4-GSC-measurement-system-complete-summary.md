## 🔧 Data Engineer の主要発見

### Search Console APIの根本的制約

> **最も重大な発見**: Search Console APIから取得できるクエリデータは、匿名化により**最大75%が欠落**する可能性がある。

**詳細**:
- プライバシー保護のため、少数ユーザーのクエリは自動匿名化
- 研究データでは46〜75%、サイトによっては93.6%が秘匿
- **これはAPI仕様であり、どの手段でも回避不可能**

**対応策**:
- 「取得可能なデータのみで分析」する前提
- ページ単位の分析を優先（クエリ単位よりデータ欠損が少ない）

### GA4データ精度の問題

| 問題 | 発生条件 | 影響 |
|------|---------|------|
| **サンプリング** | Exploration使用時、1,000万イベント超 | データが推定値になる |
| **Thresholding** | Google Signals有効、少数ユーザー | データが隠される |
| **カーディナリティ制限** | ページ数が5万超 | ロングテールが(other)に集約 |

**対応策**:
- 標準レポート範囲での分析（Exploration回避）
- BigQuery Export（Phase 3で検討）

## 📊 Analytics QA の主要発見

### Agent Teams 設計の問題点

**結論**: 元々提案されたAgent Teams構想（5エージェント）は実際の提案書には含まれていなかった。

**理由**:
1. **月10回の頻度では自動化ROIが低い**: 人間がSlackレポートを読む方が確実で速い
2. **LLMの確率的ブレが因果推論に不適合**: 数値計算はGASの確定的ロジックで行うべき
3. **人間の判断が必要**: 外部要因の解釈、施策判断は専門知識が不可欠
4. **保守が困難**: プログラマーでないユーザーにとってプロンプト調整が難しい

### 人間の介入ポイント（必須）

| 段階 | 介入が必須な状況 | 判断基準 |
|------|---------------|---------|
| バナー差替え時 | Googleフォームに入力 | 差替え実施後、必ず入力 |

## ⚠️ Risk Assessor の主要発見

### 「沈黙の失敗」最悪シナリオ

複合障害により、**気づかずに誤ったデータで分析してしまう**リスク：

1. GA4のThresholdingでデータが一部隠される
2. GSCが50時間以上遅延中だが、キャッシュで古いデータ表示
3. Googleのアルゴリズムアップデート進行中
4. API制限超過で一部データ取得失敗、古いキャッシュで補完
5. この「古い・不完全・歪んだ」データで分析
6. 誤った結論に基づいて大規模投資

**発生確率**: 低くない（個別リスクはそれぞれ「中〜高」）

### 最重要対策（5項目）

1. **全レポートに「データ取得日時」を必ず表示** — 多くのリスクを検知可能
2. **GA4のBigQuery Export を有効化** — Phase 3で検討（サンプリング問題を根本解決）
3. **GA4/GSCの差異率の「正常範囲」を事前定義** — 60-80%の差は正常
4. **データ取得失敗時はレポート生成を停止** — 古いデータでの分析を防ぐ
5. **人間の最終確認を必須化** — レポートは「速報値」、判断は人間

### ステップ3: Config.gs の作成

取得した2つの値（SPREADSHEET_ID、SLACK_WEBHOOK_URL）を使って、Config.gsを作成：

```javascript
// Config.gs - 設定ファイル

// GA4 設定
const GA4_PROPERTY_ID = '356839446';

// Search Console 設定
const GSC_SITE_URL = 'https://jo-katsu.com/';  // または sc-domain:jo-katsu.com

// スプレッドシート設定
const SPREADSHEET_ID = '【ステップ1で取得したスプレッドシートID】';

// Slack 設定
const SLACK_WEBHOOK_URL = '【ステップ2で取得したWebhook URL】';

// 計測設定
const MEASUREMENT_CONFIG = {
  beforeDays: 14,        // Before期間（日数）
  afterDays: 14,         // After期間（日数）
  gscDelayDays: 3,       // GSCデータ取得の安全マージン（日数）

  // 判定閾値
  thresholds: {
    significant: 0.05,   // ±5% で「改善」「悪化」と判定
    reportDay: 17        // 変更後17日目にレポート送信
  }
};

// 計測指標の重み（スクロール指標を除外した12指標版）
const METRIC_WEIGHTS = {
  // Tier 1: トラフィック指標（50%）
  sessions: 0.20,           // セッション数
  ctr: 0.30,                // CTR（GSC）

  // Tier 2: エンゲージメント指標（30%）
  avgEngagementTime: 0.10,  // 平均エンゲージメント時間
  engagementRate: 0.10,     // エンゲージメント率
  bounceRate: 0.10,         // 直帰率

  // Tier 3: コンバージョン指標（20%）
  conversions: 0.20         // コンバージョン数
};

// その他の監視指標（重みなし）
const MONITORING_METRICS = [
  'pageviews',              // ページビュー数
  'users',                  // ユーザー数
  'newUsers',               // 新規ユーザー数
  'clicks',                 // クリック数（GSC）
  'impressions',            // 表示回数（GSC）
  'position'                // 平均掲載順位（GSC）
];
```

**重要**:
- `【ステップ1で取得したスプレッドシートID】`を実際のIDに置き換える
- `【ステップ2で取得したWebhook URL】`を実際のURLに置き換える

## 🔄 運用フロー

### 1. バナー変更時（所要: 30秒）

```
バナー差替え実施
    ↓
Googleフォームに記事URLを入力
    ↓
送信
    ↓
自動的にスプレッドシートに記録
    ↓
GASが17日後のトリガーを設定
```

### 2. 17日後（自動実行）

```
GASが自動実行（深夜1時）
    ↓
Before期間（14日間）のデータ取得
    ↓
After期間（14日間）のデータ取得
    ↓
12指標を比較
    ↓
重み付きスコアを計算
    ↓
判定（改善/悪化/変化なし）
    ↓
Slackに結果を通知
```

### 3. レポート確認と判断（所要: 5分）

**Slackレポートの例**:

```
【17日後レポート】記事A - バナー差替え効果測定

■ 総合判定：改善 ✓
■ 重み付きスコア：+8.2%

【詳細】Tier 1（トラフィック）
- セッション数：100 → 120 (+20%)
- CTR（GSC）：3.5% → 4.2% (+0.7pt)

Tier 2（エンゲージメント）
- 平均エンゲージメント時間：45秒 → 52秒 (+15.6%)
- エンゲージメント率：65% → 70% (+5pt)
- 直帰率：45% → 40% (-5pt)

Tier 3（コンバージョン）
- コンバージョン数：5 → 8 (+60%)

その他の指標
- ページビュー数：150 → 180 (+20%)
- ...
```

**判断基準**:

| 重み付きスコア | 判定 | ネクストアクション |
|--------------|------|----------------|
| **+5%以上** | 🟢 改善 | ✓ バナー変更は成功<br>✓ 同様のデザインを他記事にも展開<br>✓ さらに改善できる要素を探す |
| **-5%〜+5%** | ⚪ 変化なし | ⚪ 効果不明<br>⚪ もう少し長期（28日後）で様子を見る<br>⚪ または大きく変える |
| **-5%以下** | 🔴 悪化 | ❌ バナーを元に戻す検討<br>❌ または別のデザインでA/Bテスト |

   - 「バナー変更の効果」と「外部要因」を分離できない
